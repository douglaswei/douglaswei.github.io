日志架构

[两层Flume日志系统](http://www.tuicool.com/articles/BNfYnum)
Flume_1 -> Flume_2 -> Kafka -> FILE/HDFS/ES -> Kibana —> Storm

日志收集

flume与logstash类似，最大的区别在于日志的格式化;
如果需要，可以在数据产生的地方结构化地封装日志;

两层Flume架构

第一层作为业务接入层，第二层作为日志缓冲层；
优点：
    1. 业务垂直切分，业务之间不收干扰，收集到的日志可按业务分割
    2. 后台的kafka升级切换的时候，不需要修改L1层，只需要修改L2层，改动对L1层透明
    3. 在L1连接L2层的时候使用failover机制

日志分层分流

通过两层Flume架构，可以做到日志按业务分割；
也可以通过定义统一的日志格式，对原始日志做分层；

CORE

概念的梳理

NRT

近实时：数据的传输，写索引，持久化，是需要时间的；

Cluster，集群

是一系列节点(服务器)组成的服务，能够分布式的建立查询索引；
使用名字来唯一区别，一个节点只能属于唯一的集群；

Node，节点

作为集群的成员，存储数据，并参与到索引的建立和查找；
使用名字唯一识别，如果未指定，使用漫画角色名标识；

Index，索引

是对于相似文档的某一列数据；

Type，类型

是一些相似文档的共性的定义，一个索引中可以有多个type；

Document，文档

文档是可以被索引的最小粒度；

Shard，分片

解决了：
当数据量太大，单一节点无法存储所有文档的场景
提高了并行计算能力

Replicas，副本

解决了
高可用性
提高了qps
